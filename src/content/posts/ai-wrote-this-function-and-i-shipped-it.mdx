The function below was written by Claude. I asked for a slug helper —
something that takes <code>"On writing fewer abstractions"</code> and
gives me <code>"on-writing-fewer-abstractions"</code> — and ten seconds
later I had this:

<CodeBlock name="slugify.ts" lang="ts" code={`export function slugify(input: string): string {
  return input
    .toLowerCase()
    .normalize('NFKD')
    .replace(/[\\u0300-\\u036f]/g, '')  // strip diacritics
    .replace(/[^a-z0-9\\s-]/g, '')      // drop punctuation
    .trim()
    .replace(/\\s+/g, '-')
    .replace(/-+/g, '-');
}`} />

Five years ago I would have written this from memory, badly, and
shipped it. Today I read it for ninety seconds before I shipped it.
The ninety seconds is the part nobody talks about.

## What ninety seconds buys you

When I review code I wrote myself, I'm checking that my plan
survived contact with the keyboard. When I review code an AI wrote,
I'm checking *whether the plan was right at all*. The
rubber duck and the typist are both me; the architect is also still me.

Concretely, here's what I'm looking for, in order, every time:

1. **Does it solve the problem I asked, or a problem
   adjacent to it?** The slugify above strips diacritics —
   great. It also drops *every* non-ASCII letter, which means
   <code>"日本語"</code> becomes the empty string. If my blog is
   English-only, fine. If it isn't, this is a silent data bug.
2. **What does it do at the edges?** Empty string,
   string of only punctuation, ten thousand characters, a string
   that's already a slug. Two of those four return <code>""</code>
   here. Is that what I want, or do I want to throw?
3. **Where does it lie about its types?** The signature
   says <code>string → string</code>. It can also return
   <code>""</code>. That's not a string in the useful sense; it's
   a sentinel. If a caller forgets, I get <code>/posts/</code>.
4. **Is it dependent on something it shouldn't be?**
   Here, no. But if the AI had reached for
   <code>lodash.kebabCase</code>, I'd want to know before I added
   a 70KB dependency for one function.

<Callout kind="note">
  The order matters. Correctness first, edges second, types third,
  dependencies fourth. I used to do dependencies first and it cost
  me a lot of time arguing about the wrong things in code review.
</Callout>

## The ninety-second test, written down

I keep a literal checklist in a sticky note on my desk. It's four
questions and it has saved me from shipping nonsense maybe a
dozen times this quarter:

<CodeBlock name="checklist.md" lang="md" code={`# Before I merge AI code

[ ] Did I run it on the inputs I actually have?
[ ] Did I run it on one input I'd never have thought to test?
[ ] Can I explain every line to a colleague?
[ ] If this breaks at 3am, can I fix it without re-prompting?`} />

The fourth one is the most important. AI is great at writing
code; it is, in my experience, mediocre at debugging code it
wrote three months ago in a context it no longer has. If you
can't fix it without the model, you don't own it.

## What I still write by hand

Roughly: anything where the *shape* of the solution is the
interesting part. Data models. Error taxonomies. The first draft
of a feature, when I'm still figuring out what it is. The boundary
between two systems. Names of things.

Anything where I already know the shape and just need the keystrokes,
the model is faster than me and roughly as accurate. Glob a directory.
Parse a CSV. Wire up a fetch with retries. Format a date in eight
locales. I am not precious about typing those out anymore.

> The skill isn't writing code. It never really was. The skill is
> knowing what code should exist, and recognizing — quickly —
> when the code in front of you is not it.

## One concrete change to your workflow

If you take one thing from this post: **read AI output
out loud, once, before you accept the diff.** Not the whole
file. Just the part that changed. Reading silently I skim; reading
aloud I notice the off-by-one, the wrong variable, the imported
thing I don't have. It feels stupid. It works.

That's the whole post. The model will write you a thousand
slugify functions this year. Read each one for ninety seconds.
That's the job.<Footnote n={1} />
